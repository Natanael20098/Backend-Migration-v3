### Existing Patterns and Conventions

- **Language**: Python 3, FastAPI, Pydantic v2, SQLAlchemy async
- **Auth**: HS256 JWT, validated exclusively via `services/shared/auth.py::verify_jwt`, PyJWT >= 2.9.0
- **Response serialization**: `CamelModel` base in `services/shared/models.py` (alias_generator = to_camel)
- **Pagination**: `PageResponse[T]` in `services/shared/models.py` — Spring Data Page envelope
- **Error format**: `{"message": "..."}` for domain errors, `{"detail": "..."}` for auth failures
- **DB**: Async SQLAlchemy, asyncpg driver, Supabase PostgreSQL (external), `prepare_threshold=0`
- **Routing**: Flat router structure; each service has `app/routers/<domain>.py`
- **Schema**: Single shared public PostgreSQL schema; snake_case column names
- **Gateway**: nginx on port 80; explicit per-location Host headers (security hardened)
- **Container**: Non-root `appuser`; HEALTHCHECK on all services

2. **Dependency rules (allowed and forbidden)**

   ```
   auth-service         → shared/
   ```

1. **End-to-end authentication flow (current Python state)**

   ```
   User → POST /api/auth/send-otp (email)
     → auth-service → rate limit check (otp_codes count last 1h)
     → if < OTP_RATE_LIMIT_PER_HOUR: generate 6-digit OTP (secrets.randbelow)
     → INSERT otp_codes (email, code, expires_at = now + OTP_EXPIRY_MINUTES)
     → Mailgun HTTP API: send OTP to email
     → 200 {"message": "If this email is registered, a code has been sent."}
     (Note: always returns same message regardless of email existence — prevents enumeration)

   User → POST /api/auth/verify-otp (email, code)
     → auth-service → SELECT otp_codes WHERE email=? AND code=? AND used=false AND expires_at > now
     → if found: UPDATE otp_codes SET used=true
     → jwt.encode({sub: email, iat: now, exp: now + JWT_EXPIRATION_MS/1000}, JWT_SECRET, HS256)
     → 200 {"token": "...", "email": "...", "expiresIn": N}

   User → Any protected endpoint (Authorization: Bearer <token>)
     → nginx gateway → FastAPI service
     → HTTPBearer security scheme (FastAPI)
     → shared.auth.verify_jwt(token, JWT_SECRET)
       → jwt.decode(token, JWT_SECRET, algorithms=["HS256"])
       → validate exp (PyJWT automatic)
       → validate sub is non-null
       → return subject (email)
     → handler receives current_user (email) as a dependency
   ```

2. **Legacy Java authentication flow (for migration comparison)**

   Java monolith flow (from `domain-boundary-map.md` §1 + `JwtAuthenticationFilter` analysis):
   ```
   Java: POST /api/auth/login (email, password) OR OTP-based flow
     → AuthController → JwtUtil.generateToken(email)
       → Jwts.builder().setSubject(email).signWith(HS256, JWT_SECRET).compact()
     → Returns {"token": "...", "expiresIn": N}

   Java: Any protected request
     → JwtAuthenticationFilter extends OncePerRequestFilter
     → Extracts Bearer token from Authorization header

     → Downstream services call SecurityUtils.getCurrentUserEmail()
   ```

   - Java does not set `iss` or `aud` → FastAPI does not validate these (intentional, per `jwt-validation-policy.md` §4) ✅

5. **Principal propagation** — how the current user is passed through the stack:
   - FastAPI: `current_user: str = Depends(get_current_user)` injects email string into route handlers

   - No role propagation in current FastAPI implementation

6. **Migration risks and compatibility needs**

   | Risk | Description | Mitigation |
   |---|---|---|

Risk is low (missing header = unauthenticated client, not normal user flow) |
