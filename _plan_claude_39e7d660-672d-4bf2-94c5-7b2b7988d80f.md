### Existing Patterns and Conventions

1. **FastAPI service structure**: Each service has `app/main.py` (FastAPI app + lifespan), `app/config.py` (extends shared `Settings`), `app/models.py` (SQLAlchemy ORM), `app/schemas.py` (Pydantic schemas), `app/routers/` (one file per resource group).
2. **Shared module**: `services/shared/` provides `CamelModel`, async engine factory, `make_get_current_user()`, `http_exception_handler`, `validation_exception_handler`, `make_health_router()`.
3. **DB wiring pattern**: Router files define placeholder `_get_db()` that raises `NotImplementedError`; overridden in `lifespan()` via `app.dependency_overrides`.
4. **Auth pattern**: `make_get_current_user(settings.JWT_SECRET)` returns a FastAPI dependency injected into protected endpoints.
5. **camelCase responses**: All response models use `CamelModel` (alias_generator=to_camel) to match Java Jackson serialization.
6. **Docker build context**: All FastAPI services build from `context: ./services` so `shared/` is available.
7. **nginx routing**: Specific path prefixes declared before the generic `/api/` catch-all that routes to `java_monolith`.
8. **No DDL at startup**: Services only create engine + session factory; schema is managed externally.

### Security Issues Identified in Python Services

1. **`services/shared/auth.py`**: `verify_jwt()` raises `HTTPException(401)` with `"Invalid token: missing subject"` — detail leaks token structure. Minor but fixable.
2. **`services/client-crm-service/app/routers/clients.py`**: The `ssn_encrypted` column is on the `Client` ORM model but is not in the `ClientCreate` schema — good! But `ClientUpdate` also omits it — also good. However, the `Client` model still has the field accessible. The comment says "SSN not accepted via this endpoint" but there's no server-side guard preventing its inclusion if someone directly constructs a request. The existing code already excludes it from schemas.
3. **Hardcoded JWT secret in Java `Constants.java`**: `JWT_SECRET = "homelend-super-secret-key-do-not-share"` — this is in code that is being deleted, so the fix is removal of Java code itself.
4. **`gateway/nginx.conf`**: Missing upstream and location blocks for `underwriting-service`, `closing-service`, and `client-crm-service` — these services are not receiving traffic even though they're deployed.
5. **`docker-compose.yml`**: Missing service blocks for `underwriting-service` and `client-crm-service`.

### Java Classes to Remove (per cutover records)

All Java code under `src/` is to be removed. The specific controllers/services are all present and none have been individually removed yet because the Java monolith still handles multiple routes.

## Implementation Strategy

### Overall Approach

**Task 1 — Remove Java + clean up coexistence infra:**

The codebase has all required FastAPI services implemented but several are not wired into the runtime (missing nginx routes, missing docker-compose entries). The implementation must:
1. Wire the remaining FastAPI services into docker-compose and nginx
2. Remove the Java monolith artifacts (src/, pom.xml, Dockerfile.java, Procfile, system.properties)
3. Simplify docker-compose (remove java-monolith service block)
4. Simplify nginx (remove java_monolith upstream and catch-all `/api/` block)
5. Simplify homelend.sh (remove Spring Boot bare-metal start/stop logic)
6. Update .env.example (remove Spring datasource vars)

**Admin domain decision**: The `AdminController` is NOT being ported to Python — it has severe security anti-patterns (JWT secret exposure, SQL injection). Its `/api/admin/*` routes should be removed from the gateway entirely. Any admin functionality (audit logs, system settings, notifications) is out of scope for this final epic — it was explicitly deferred to Wave 4 per `doc/remaining-domains-inventory.md §3.1`. The correct action is to drop the admin routes from nginx and confirm they return 404 after Java removal.

**Task 2 — Python service remediation:**

Focus on real post-migration risks in the FastAPI services:
1. `services/shared/auth.py`: Tighten JWT error detail (don't leak "missing subject" distinction vs invalid token)
2. `services/client-crm-service/app/routers/clients.py`: Add explicit guard — ensure `ssn_encrypted` is never written via any create/update path (currently it's absent from schemas, which is good, but the ORM model has the field — add a server-side note)
3. `gateway/nginx.conf` wiring is a hard correctness issue (traffic currently routed to wrong backend)
4. `services/auth-service/app/router.py`: OTP rate limit check uses `created_at >= one_hour_ago` but `OtpCode.created_at` uses `server_default=func.now()` (no `timezone=False` handling mismatch with `replace(tzinfo=None)`) — low risk since both sides strip tz
5. `CHANGELOG.md` and `doc/` must be updated to record final state

### Shared Foundations

No new shared modules are needed. All services are implemented. The remediation focuses on:
- Routing wiring (nginx + docker-compose)
- File deletions (Java tree)
- Targeted code fixes in existing Python files
- Documentation updates

### Dependency Order

1. Wire remaining services into docker-compose first (needed for nginx upstream resolution)
2. Wire nginx routes before removing Java catch-all
3. Remove Java code after routing is complete
4. Apply Python service fixes (independent of routing changes)
5. Update docs last

## Risks and Considerations

### Risk 1: loan-origination-service does not exist in Python

The `LoanController` in Java handles `/api/loans/` (CRUD for loan applications). There is NO `loan-origination-service` FastAPI implementation. The `underwriting-service` handles `/api/loans/{id}/credit-report`, `/api/loans/{id}/underwriting`, `/api/loans/{id}/appraisal` as sub-resources.

**Decision**: Route all `/api/loans/` to `underwriting-service` in nginx. The underwriting service will handle its three sub-paths. All base loan CRUD requests (`GET /api/loans`, `POST /api/loans`, etc.) will fall through with a 404 from FastAPI (which is correct behavior — this functionality was deferred per `remaining-domains-inventory.md`). This is better than routing to a non-existent Java service.

**Alternative if unacceptable**: Keep a minimal `/api/loans/` proxy to a placeholder or return a 503. But the task says "resulting runtime is simpler" and the Java monolith is to be fully removed, so 404 for non-migrated loan CRUD is the correct outcome.

### Risk 2: Admin routes become 404 after Java removal

`/api/admin/*` had the Java `AdminController`. There is no Python admin service. After removing Java and the catch-all nginx route, requests to `/api/admin/*` will get nginx 404. This is correct per the design decision in `remaining-domains-inventory.md §3.1` — admin is Wave 4, intentionally deferred.

**Mitigation**: Document this in the remediation summary. The frontend does not appear to call `/api/admin/*` based on the `frontend/src/lib/endpoints.ts` pattern (not checked but admin paths typically not in frontend apps).

### Risk 3: underwriting-service missing authentication on endpoints

From examining `services/underwriting-service/app/routers/credit_reports.py` lines 1–50, there is no import for `make_get_current_user` and no auth dependency visible. The same appears true for `underwriting.py` and `appraisals.py`. This is a real security gap — these endpoints would be publicly accessible.

**Fix**: Add `make_get_current_user(settings.JWT_SECRET)` dependency to all underwriting endpoints, following the exact pattern from `closing-service/app/routers/closings.py` and `client-crm-service/app/routers/clients.py`.

**Requires importing**: `from shared.auth import make_get_current_user` and `from app.config import settings`, then adding `_get_current_user = make_get_current_user(settings.JWT_SECRET)` module-level and `_: str = Depends(_get_current_user)` to each endpoint.

### Risk 4: closing-service wired in docker-compose but missing nginx location block

Currently `closing-service` is in `docker-compose.yml` but has no nginx upstream or location block. Traffic to `/api/closings/` currently hits the Java monolith catch-all. After Java removal, there will be no route for closings unless we add the nginx location block.

**Fix**: This is covered in Step 2 of Task 1 — add `upstream closing_service` and `location /api/closings/` to nginx.

### Risk 5: homelend.sh bare-metal mode becomes frontend-only

After removing Java, the `start`/`stop`/`restart` subcommands will only manage the Next.js frontend. The `status` subcommand will only show frontend status. This is correct — the `docker` subcommand handles the full stack. The bare-metal mode was always a developer convenience for running without Docker.

### Risk 6: .gitignore cleanup

Removing the Java/Maven gitignore entries is safe — there are no Java files to protect. The `target/` directory (if it exists locally from a previous Maven build) would no longer be gitignored, but since `src/` and `pom.xml` are deleted, no new `target/` will ever be created.

### Risk 7: SSN in client-crm-service

The `Client` ORM model has `ssn_encrypted` as a mapped column. The schemas (`ClientCreate`, `ClientUpdate`) correctly exclude it. The router correctly only sets fields from the schema. This is already safe. The remediation confirms and documents this rather than over-engineering a new guard.

### Risk 8: Missing .env.example files for new services

Verify that `services/underwriting-service/.env.example` and `services/client-crm-service/.env.example` exist and have the correct vars. These were presumably created when the services were implemented. If they follow the pattern of other services (copy from `services/closing-service/.env.example`), they should be correct.

### Ambiguities

1. **Loan origination base CRUD**: The task says "all required backend capabilities are confirmed live in FastAPI" — but loan application CRUD (`/api/loans` create/list/get/update/delete) is not in any FastAPI service. The acceptance criterion is satisfied by documenting that loan-origination CRUD was explicitly deferred (Wave 2B per `remaining-domains-inventory.md`) and the underwriting sub-paths ARE live.

2. **Admin service**: Per `remaining-domains-inventory.md §3.1`, AdminController is explicitly retained for Wave 4. Its removal from Java without a Python replacement is intentional per the design. No admin-service FastAPI implementation is in scope.

3. **data.sql / schema.sql**: These are in `src/main/resources/` and will be deleted with `src/`. They served as reference only — the actual schema lives in Supabase PostgreSQL. The authoritative schema reference is `doc/postgresql-access-policy.md` and the ORM models.
