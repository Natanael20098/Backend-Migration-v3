## Codebase Analysis

```

├── auth-service/
```

### Existing Patterns and Conventions
- **Shared auth module**: `services/shared/auth.py` is the single location for JWT validation logic (`verify_jwt` + `make_get_current_user`). All non-auth services delegate to this — they never call `jose` directly.
- **Auth-service only**: `services/auth-service/app/router.py` is the one place that also imports `jose` directly (for `jwt.encode` at token issuance).
- **Algorithm hardcoded to HS256**: Both `shared/auth.py` and `auth-service/app/router.py` hardcode `algorithm="HS256"` / `algorithms=["HS256"]` — not read from input.
- **Expiration check**: python-jose automatically validates `exp` when decoding. PyJWT 2.x does the same.
- **Error handling**: Both files catch `JWTError` and raise `HTTPException(401)` — this maps directly to PyJWT's `jwt.PyJWTError`.
- **Token shape**: `sub` (email), `iat`, `exp` — no audience or issuer claims currently used.
- **Documentation style**: `doc/` directory holds markdown policy files (api-compatibility-rules.md, postgresql-access-policy.md, etc.) — JWT policy doc should follow the same convention.
- **requirements.txt**: All 5 services pin `python-jose[cryptography]==3.3.0`. The `shared/` module is not packaged — it is copied into each Docker container. Non-auth services do not have their own `requirements.txt` entries for `jose`; they inherit the dependency by listing it redundantly.
- **Tests**: There are currently **no test files** anywhere in the project. Task 3 requires creating tests.

### Vulnerability Profile
- `python-jose 3.3.0` has **CVE-2022-29217**: algorithm confusion attack where the library could accept tokens signed with an unexpected algorithm when `algorithms=` is not specified.
- The existing code **does** specify `algorithms=["HS256"]` explicitly — this specific CVE vector is already mitigated in practice, but the pinned version is still considered vulnerable and should be replaced.
- `python-jose 3.5.0` is the latest release but the library is largely unmaintained.
- **Replacement decision**: Migrate to `PyJWT==2.9.0` (stable, actively maintained, widely adopted). PyJWT 2.x:
  - `jwt.encode(payload, secret, algorithm="HS256")` → returns `str` (no `.decode()` needed).
  - `jwt.decode(token, secret, algorithms=["HS256"])` → same call signature.
  - `jwt.PyJWTError` replaces `jose.JWTError` as the base exception.
  - Validates `exp` automatically, raises `jwt.ExpiredSignatureError` (subclass of `jwt.PyJWTError`).
  - Does **not** need the `[cryptography]` extra for HS256 — no extra dependencies.

## Implementation Strategy

### Overall Approach
1. **Migrate the shared library first** (`services/shared/auth.py`) — this single change benefits all non-auth services simultaneously since they all delegate to it.
2. **Migrate auth-service second** — update `router.py` (the only service doing token issuance) and its `requirements.txt`.
3. **Update non-auth service requirements.txt** — replace `python-jose` with `PyJWT` in client-crm, closing, underwriting. Remove JWT library entirely from property-listing.
4. **Create tests for auth-service** — new `tests/` directory with pytest covering encode/decode and invalid-token paths.
5. **Write the JWT policy document** — `doc/jwt-validation-policy.md`.
6. **Update CHANGELOG**.

### Dependency Order
```
Task 1 (policy doc) — independent, no code deps
Task 2 (inventory) — already analyzed; manifest as doc section in policy + plan
Task 3 (upgrade auth-service) — depends on shared migration being done first
Task 4 (remove unnecessary deps) — independent of Task 3 since they use shared.auth
```

### Shared Foundation
`services/shared/auth.py` is the single shared foundation — migrating it from `python-jose` to `PyJWT` cascades to all consumer services without touching their router code.

### PyJWT vs python-jose API Delta (one-to-one mapping)
| python-jose | PyJWT 2.x |
|---|---|
| `from jose import JWTError, jwt` | `import jwt` |
| `jwt.decode(token, secret, algorithms=["HS256"])` | `jwt.decode(token, secret, algorithms=["HS256"])` ← identical |
| `jwt.encode(claims, secret, algorithm="HS256")` | `jwt.encode(claims, secret, algorithm="HS256")` ← identical |
| `except JWTError:` | `except jwt.PyJWTError:` |

## Execution Plan (by phase)

### BUILD Phase

#### Step 1 — JWT Validation Policy Document (Task 1)
- [ ] File: `doc/jwt-validation-policy.md` — **CREATE**
  - Documents the official JWT validation policy for all Python services.
  - Sections:
    - **Algorithm**: HMAC-HS256 only; algorithm is never read from token header or request input; `algorithms=["HS256"]` is always passed explicitly to the decoder.
    - **Library**: PyJWT ≥ 2.9.0 (replaces python-jose 3.3.0 CVE-2022-29217).
    - **Claims validated**: `exp` (expiration) — validated automatically by PyJWT on every decode; `sub` (subject/email) — must be present and non-null; `iat` — included in issued tokens.
    - **Issuer/Audience**: Not used currently (compatible with Java jjwt tokens); not validated.
    - **Error behavior**: Any `jwt.PyJWTError` (including `ExpiredSignatureError`, `DecodeError`, `InvalidSignatureError`) → HTTP 401 with `{"message": "Invalid or expired token"}` (or `{"error": ...}` for auth paths per existing exception handler convention).
    - **Where validation occurs**: Only in `services/shared/auth.py::verify_jwt`. Services must not implement local JWT validation; they import `make_get_current_user` from `shared.auth`.
    - **Token issuance**: Only `auth-service` issues tokens. No other service may call `jwt.encode`.
    - **Secret**: Shared `JWT_SECRET` env var (minimum 32 chars recommended); must match across all services.

## Risks and Considerations

### 1. PyJWT `jwt.encode` return type
In `python-jose`, `jwt.encode()` returns `str`. In `PyJWT 1.x` it returned `bytes`, but **PyJWT 2.x returns `str`** — same as python-jose. No `.decode("utf-8")` call is present in `router.py`, so no change needed there.

### 2. Property-listing-service has no auth but has python-jose in requirements
This is confirmed as a copy-paste artifact. Removing `python-jose` and **not** adding PyJWT is safe — no import in that service touches JWT.

### 3. Shared module is not a separate package
`services/shared/` is copied directly into each Docker image. Therefore each service's own `requirements.txt` must include `PyJWT` if `shared.auth` is imported (because shared.auth will `import jwt`). The exception is `property-listing-service` which does not import `shared.auth`.

### 4. No `cryptography` extra needed for PyJWT + HS256
python-jose required `[cryptography]` for RSA/EC algorithms, but used it even for HS256. PyJWT 2.x uses only Python stdlib (`hmac`, `hashlib`) for HS256. No `cryptography` extra is needed. This reduces the dependency surface.

### 5. CVE-2022-29217 was practically mitigated
The existing code already specified `algorithms=["HS256"]` explicitly, which is the primary mitigation for the algorithm confusion CVE. The upgrade is still required because: (a) the pinned version is flagged as vulnerable by scanners, (b) the library is poorly maintained, (c) PyJWT is the industry standard.

### 6. Test infrastructure
No pytest or test dependencies are currently in any requirements.txt. The tests created in Task 3 use only `pytest` which must be runnable. The `pytest.ini` approach sets `pythonpath = ..` relative to `services/auth-service/`, which resolves to `services/` — making `shared` importable. If the CI/CD environment installs from requirements.txt only, a separate `requirements-dev.txt` or `pytest` listed in main `requirements.txt` would be needed. Since the task says "tests cover encode/decode and invalid token cases" and there is no existing CI setup, the plan adds `pytest` to the auth-service requirements.txt alongside PyJWT.

### 7. No integration test change needed
Non-auth services don't need test changes — they use `make_get_current_user` from `shared.auth` unchanged. The function signature, behavior, and HTTP response codes are all identical after the migration.

### 8. Task 2 (inventory) is a documentation/analysis task
The inventory findings are documented in this plan and in the JWT policy document. No separate "inventory report" file is required — the policy doc's "per-service strategy" section fulfills the acceptance criteria ("Propose a per-service strategy: upgrade, remove, or centralize").
