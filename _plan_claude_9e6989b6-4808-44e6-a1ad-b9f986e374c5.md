### The Security Issue (Audit Findings for Task 1)

**Location:** `gateway/nginx.conf`, line 68:
```nginx
proxy_set_header Host  $host;
```

**What `$host` contains in nginx:**
- Primary: the value of the client-supplied `Host` request header (`$http_host`)
- Fallback: the `server_name` directive value if no `Host` header is present or if it is empty
- This means an attacker who controls the `Host` header controls the value forwarded to all upstreams

**Why this is a problem:**
1. **HTTP Host header injection / cache poisoning**: If any upstream generates absolute URLs (e.g., for redirects, password reset links, OAuth callbacks), it may use the forwarded `Host` value, allowing an attacker to redirect victims to attacker-controlled domains.
2. **Password-reset / email link poisoning**: The auth-service sends OTP emails via Mailgun. If it ever uses the forwarded Host to construct a callback URL (a common pattern), those links could be poisoned.
3. **Unvalidated Host forwarding breaks defense-in-depth**: Even if current services don't use the Host value, hardened infrastructure should not forward attacker-supplied values.
4. **Scope**: All `location` blocks inherit the server-level `proxy_set_header Host $host;` directive — auth, property-listing, underwriting, closing, client-crm, **and** the frontend passthrough (`/`).

**Routes depending on current behavior:**
- `/api/auth/` → `auth-service:8001` — inherited Host forwarding, no current usage detected in Python code
- `/api/properties/`, `/api/listings/` → `property-listing-service:8002` — same
- `/api/loans/` → `underwriting-service:8003` — same
- `/api/closings/` → `closing-service:8004` — same
- `/api/clients/`, `/api/agents/`, `/api/brokerages/`, `/api/leads/`, `/api/showings/`, `/api/offers/` → `client-crm-service:8005` — same
- `/` → `frontend:3000` — inherits; Next.js uses the Host header for URL construction in SSR, so this case requires special treatment (see below)

**Safe Replacement Strategy:**
Use `$http_host` allowlist validation via an nginx `map` directive. This replaces the unchecked `$host` with a validated value:
- Define an explicit allowlist of valid Host values using an nginx `map` block
- If the incoming `Host` matches an allowed value, forward it as-is
- If the incoming `Host` does NOT match, replace it with a hardcoded safe default (`localhost`)
- For the frontend (`/` location), keep forwarding a valid Host so Next.js SSR works correctly; the validated (not attacker-supplied) value is used

**Why not hardcode a single Host value:**
The deployment uses `localhost` in local dev and will use a real domain in production. Using a map/allowlist is the canonical nginx pattern that works across environments and doesn't break Next.js SSR or cookie/redirect logic.

### Dependencies and Tech Stack
- **nginx 1.25-alpine** (Docker image) — supports `map`, `geo`, `valid_referers` directives natively
- **Docker Compose 3.9** — bridge network `platform`
- **FastAPI** (Python) — services don't read Host; they use JWT for auth
- **Next.js 14** — frontend uses `NEXT_PUBLIC_API_URL` env var baked at build time; SSR pages may use `req.headers.host` for URL construction

## Implementation Strategy

### Overall Approach
1. **Task 1** is an audit + documentation task. Its deliverable is a section in `doc/` (or inline in the plan) documenting findings, affected routes, and the chosen strategy. Given the project convention of putting security policies in `doc/`, a short audit note in `gateway/README.md` is appropriate (the README already has a "Configuration Notes" section). Alternatively, add a `doc/gateway-host-header-audit.md`. The cleanest fit per project convention: add an **Audit & Security** section to `gateway/README.md`.
   
2. **Task 2** is the actual fix: modify `gateway/nginx.conf` to use an allowlist `map` instead of blindly forwarding `$host`.

### Shared Foundations
- The `map` block goes in the `http {}` context of `nginx.conf`, before the `server {}` block.
- The allowlist covers known valid host values: `localhost`, `localhost:80`, `127.0.0.1`, `127.0.0.1:80`, and a placeholder for a production domain.
- The safe fallback value is `localhost` (i.e., the server's own name).

### Dependency Order
1. First: write the audit documentation (Task 1 deliverable) in `gateway/README.md` — no code dependency
2. Second: modify `gateway/nginx.conf` with the map + validated forwarding (Task 2) — references the strategy from Task 1
3. Third: update `CHANGELOG.md`

#### Task 2 — Fix `gateway/nginx.conf`

- [ ] File: `gateway/nginx.conf` — **MODIFY** — Apply the following changes:

  **Step 1 – Add a `map` block** in the `http {}` context, before the `server {}` block:
  ```nginx
  # ── Host header allowlist ────────────────────────────────────────────────────
  # Replace attacker-controlled $host with a validated value.
  # If the incoming Host matches an allowed value, forward it unchanged.
  # If it does not match (or is absent), forward the safe fallback "localhost".
  # Extend this list when deploying to a real domain.
  map $http_host $validated_host {
      default           "localhost";
      "localhost"       "localhost";
      "localhost:80"    "localhost:80";
      "127.0.0.1"       "127.0.0.1";
      "127.0.0.1:80"    "127.0.0.1:80";
  }
  ```

  **Step 2 – Replace the insecure directive** in the `server {}` block:
  ```nginx
  # BEFORE (insecure — forwards attacker-controlled value):
  proxy_set_header Host  $host;
  
  # AFTER (safe — forwards only allowlisted values, falls back to "localhost"):
  proxy_set_header Host  $validated_host;
  ```

  The rest of the server block (all `location` blocks, all other `proxy_set_header` directives) remains unchanged. All routes continue to function because:
  - FastAPI microservices don't use the `Host` header for routing or logic
  - The frontend Next.js container receives a validated host value (which it uses for SSR URL construction)
  - Legitimate clients (`localhost`, `localhost:80`) continue to pass validation unmodified

## Risks and Considerations

### Potential Conflicts Between Tasks
- Tasks 1 and 2 are sequential: Task 1 documents the strategy that Task 2 implements. They touch the same files (`gateway/nginx.conf`, `gateway/README.md`) but in non-conflicting ways.
- No conflicts with other services — no Python service reads the `Host` header.

### Next.js Frontend Consideration
The `/` location proxies to `frontend:3000`. Next.js in SSR mode may use `req.headers.host` to construct absolute URLs. With the `map` allowlist:
- In local development with `Host: localhost`, the frontend receives `localhost` — correct.
- With an invalid/missing Host, the frontend receives `localhost` — still safe and functional.
- When deploying to a real domain (e.g., `app.homelend.io`), that value must be added to the `map` block. The README audit section should explicitly call this out.

### WebSocket / Hot Reload (Development)
The frontend location block has:
```nginx
proxy_set_header Upgrade    $http_upgrade;
proxy_set_header Connection "upgrade";
```
These are unaffected by the Host header change. WebSocket upgrades continue to work.

### The `$host` vs `$http_host` Distinction
- `$http_host` = exactly what the client sent in the `Host` header (including port if present)
- `$host` = `$http_host` normalized (lowercase, port stripped for standard ports), or `server_name` if no Host header
- The `map` uses `$http_host` as the input (raw client value) to catch injection attempts. The output `$validated_host` is always a known-safe string.

### Missing Information / Ambiguities
- **Production domain**: The allowlist in the `map` block needs a real production hostname when the platform is deployed publicly. The plan includes a comment in the nginx config directing operators to extend the list.
- **`server_name _`**: The current config uses `server_name _` (catch-all). This is intentional for local dev. No change needed here — the Host header allowlist is independent of `server_name`.
- **No HTTPS yet**: The config only listens on port 80. When HTTPS is added (port 443, TLS termination at nginx), the allowlist will need updating. This is out of scope for this epic.

### Integration Points That Need Care
- **`gateway/README.md` sync**: The README routing table and smoke tests must stay in sync with `nginx.conf`. The plan includes README updates alongside every nginx change.
- **`CHANGELOG.md`**: Per project convention, every change gets a CHANGELOG entry. The plan includes this explicitly.
- **No service restarts needed**: The FastAPI services are unchanged. Only the gateway container needs to be rebuilt/reloaded (`docker compose up --build gateway` or `nginx -s reload` inside the container).
